Terminology

  • a11y = accessibility
  • AT = assistive technology
  • Screen reader = software that interprets code and reads content
  • WAI-ARIA = W3C technical spec, enables modern a11y
  • WCAG2 = W3C a11y guidelines, v2
  • Rehabilitation Act(1973), Section 508 = provides electronic a11y guidelines for US government vendors
  • Americans w/Disabilities Act (ADA, 1990) = law, prohibits discrimiation

A11y is important

  • Companies: Legal and brand risks
  • People w/disabilities: Independence, dignity
  • A11y benefits everyone:
    • temporary disabilities, curb cuts, door openers
    • accessible web = machine-readable web = voice control is the future...

Modern Web Challenges

What makes accessible, modern web UI so challenging?

  • Modern web UI is inherently inaccessible
    • Dynamic UI: content updates, state changes not communicated
    • Custom UI: role and purpose of non-native widgets
  • ARIA attributes help bridge this gap

Design/Development Tips

  • Create elegant, stable technical solutions
  • Minimize use of ARIA - rely on semantic code
  • Aim for widest screen reader/browser support possible
  • Test often - don't wait until the end
  • Developers and QA peeps should be screen reader experts
  • Make accessibility easier, repeatable
  • Build a11y into your process
  • Accessibility is a team effort
  • Be wary of "accessible" frameworks - test for yourself, BYO
  • Mimic native behaviors

Tools and Techniques

Focus Management

Managing where focus is and avoiding the loss of focus

Tips:

  • Focusing on an element reads its contents or label
  • Use carefully - don't move focus unless necessary
  • Only natively focusable or tabindexed elements can receive focus
  • Hard coding tabindex can be problematic --> do not hard code tabindex
  • Use a script to dynamically manage focus…

Focus on anything w/$.access()

How it works:

  • Adds tabindex="-1", places focus and cleans up on focusout

Example Usage:

  • Focused item is no longer available
  • Modals
  • SPA view updates

One Live Region To Rule Them All

Live region pitfalls:

  • Multiple live regions cumbersome to manage
  • Live region roles(status, alert, log) don't work reliably
  • Ad-hoc live region use adds complexity, risk

Best practices:

  • Use only ONE live region in your entire site/app
  • Use a script to announce content…

Implementing A11y

"Semantic" Markup

Semantic markup is the foundation of a11y, but...

  • Generic tags are sometimes better, less verbose
  • Most important: How is markup spoken?
  • Examples:
    • HTML5 sectioning elements spoken as landmarks
    • Buttons and anchors vs. generic elements
    • UL/LI for true list content ONLY
    • DL/DD/DT are overly verbose

Headings

Headings are crucial to page navigation and content hierarchy

  • UX team should determine heading strategy
  • Headings describe sections or groups of content
  • Order should not increase by more than 1
  • Don't overuse
  • Don't put links/buttons in headings

Landmarks

Landmarks improve navigation and understanding of page structure

Role HTML Tag Usage Notes
banner <header role="banner"> once per view Page header
main <main role="main"> once per view Main content area
contentinfo <footer role="contentinfo"> once per view Page footer
search - multiple Search form
form - multiple Don't use unless a form plays a significant role in page
navigation <nav role="navigation"> multiple Use with label/heading
complementary <aside role="complementary"> multiple Use with label/heading
application - !avoid

Landmark Usage

At a minimum, all pages should have:

  • banner - <header role="banner">
  • main - <main role="main">
  • contentinfo - <footer role="contentinfo">

Goal: ensure that all content resides in a landmark region and, in doing so, no content is orphaned

Landmark Tips

  • Create a landmark strategy, use it consistently
  • Use labels for generic landmarks (e.g. navigation, complementary)
  • Some HTML5 sectioning elements are landmarks by default
  • Use HTML5 element and role
  • Don't overuse

Modals

Creating accessible modals

Modal setup:

  • start heading order with H1
  • use aria-labelledby to label modal
  • add role=dialog to container

On open:

  • focus on first focusable or dialog body
  • isolate modal UI from view content
  • dialog is announced

On close:

  • return focus to button, last focused or main content
