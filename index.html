Content Navigation

  • How do people understand your content non-visually?
  • Content hierarchy
  • How do people navigate page content?

Headings

Headings improve navigation and establish page hierarchy

  • Order should not increase by more than 1
  • One H1 per view
  • Don’t overuse - H1 to H4 is good
  • Use headings to describe sections or groups of content
  • Don’t put links/buttons in headings

Landmark Tips

  • Create a landmark strategy, use it consistently

  • Don’t overuse - depends on page complexity, though...

Tips:

  • Use a script to manage focus

One Live Region To Rule Them All

Live region pitfalls:

  • Multiple live regions cumbersome to manage
  • Live region roles(status, alert, log) don't work reliably
  • Ad-hoc @aria-live use adds complexity, risk

Best practices:

  • Use only ONE @aria-live in each page
  • Use a script to speak content

Say Anything w/$.announce

How it works:

  • Create common @aria-live “announcer” element
  • Announcer needs to be in DOM on page load and left alone
  • Reset technique allows repeat announcements

Inline Forms Validation

Inline, dynamic errors problematic

Tips:

  • Leverage field label to report error messaging
  • Use $.announce to speak errors
  • Use ARIA to communicate states:
    • @aria-invalid, @aria-required

Alternate Copy and Labeling

Additional/alternate copy used to improve accessibility

Examples: button/link copy, form errors, visual translations

Implementation:

  • aria-label=“{label text}” - overrides content
  • aria-labelledby=“{1 or more container id’s}” - label an element using any copy in the view
  • aria-describedby=“{1 or more container id’s}” - additional descriptive content
  • title behavior is not as consistent

Single Page Architecture

No page refresh - users have no indication of context changes

Solution: $.announce "view loading/loaded" status

  • Update page title from view to view
  • Use $.announce to speak "{new title} view loaded"
