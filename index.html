Introduction

Terminology

  • a11y = accessibility
  • assistive technology = AT
  • primary screen readers
    • JAWS, NVDA, WindowEyes - Windows
    • VoiceOver - MacOS, iOS
    • TalkBack - Android
  • WAI-ARIA
  • WCAG2
  • Rehabilitation Act, Section 508
  • Americans w/Disabilities Act (ADA)
  • Device-independence - works w/keyboard and mouse

Accessibility is important

  • ADA mandates equal access
  • (avoid lawsuits, risk mitigation)
  • Benefits everyone
    • accessible = machine-readable = voice control is the future...
    • temporary disabilities, curb cuts, door openers

Standards compliance vs. true accessibility

  • "Standards compliant" is not good enough
  • Make experiences user-friendly for people of all abilities

Tools and Techniques

"Semantic" Markup

  • Semantic markup and logical hierarchy are the foundation
  • Generic is sometimes better
  • Buttons vs. anchors vs. generic elements
  • Some HTML5 sectioning elements spoken as landmarks
  • More important: how is markup spoken?

Content Navigation

  • How do people understand your content non-visually?
  • Content hierarchy
  • How do people navigate page content?

Headings

Headings improve navigation and establish page hierarchy

  • Order should not increase by more than 1
  • One H1 per view
  • Don’t overuse - H1 to H4 is good
  • Use headings to describe sections or groups of content
  • Don’t put links/buttons in headings

Landmark Tips

  • Use labels for generic landmarks (e.g. navigation, complementary)
  • Some HTML5 sectioning elements are landmarks by default
  • Use HTML5 element AND role
  • Don’t overuse - depends on page complexity, though...
<section role="complementary" aria-labelledby="module_heading">
	<h2 id="module_heading">Frequently Asked Questions</h2>
	...
<header role="banner">
<footer role="contentinfo">

Focus Management

Managing where focus is and avoiding the loss of focus

  • modals
  • view updates

Tips:

  • Focusing on an element either reads the content or any label(if available)
  • Use carefully - don’t move focus unless necessary
  • Only natively focusable or tabindex
  • Use a script to manage focus

Announcing Content

@aria-live creates "live region"

Updates to live regions are automatically spoken by the screen reader

Usage:

  • alerts
  • errors
  • notifications
  • significant/pertinent updates

Announcing Content

@aria-live pitfalls:

  • Multiple live regions cumbersome to manage
  • adds complexity

Best practices:

  • Avoid ad-hoc use of @aria-live
  • Use script to speak content

Say Anything w/$.announce

How it works:

  • Create common @aria-live “announcer” element
  • Announcer needs to be in DOM on page load
  • Reset technique allows repeat announcements
